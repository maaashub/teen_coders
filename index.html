Day 2: Building Blocks of the Web!

What is HTML?

HTML stands for HyperText Markup Language.

  • It's NOT a programming language, but a markup language.
  • It uses "tags" to define the content and structure of a web page.
  • Think of it as the blueprint or skeleton of your website.
  • Every webpage you visit is built with HTML!

HTML Tags: The Building Blocks

Opening and Closing!

HTML uses "tags" to mark up elements. Most tags come in pairs:

  • An opening tag: <tagname>
  • A closing tag: </tagname>
  • The content goes between the tags.

Example:

<p>This is a paragraph.</p>

Some tags are "self-closing" or "void tags" (they don't need a closing tag), like <br> for a line break.

Basic HTML Document Structure

The Essential Blueprint

Every HTML page starts with a basic structure:


<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>My First Webpage</title>
</head>
<body>
    <!-- Content goes here -->
</body>
</html>
  • <!DOCTYPE html>: Declares the document type.
  • <html>: The root element of an HTML page.
  • <head>: Contains meta-information (not visible on the page).
  • <body>: Contains all the visible content of the page.

Common HTML Elements: Headings & Paragraphs

Organizing Your Text

  • Headings (<h1> to <h6>): Define titles and subtitles. <h1> is most important.
    • <h1>Main Title</h1><h2>Subtitle</h2>
  • Paragraphs (<p>): For blocks of text.
    • <p>This is a sentence.</p>
  • Line Breaks (<br>): Inserts a single line break. (Self-closing).
    • Line 1.<br>Line 2.

Day 3: Organizing Content with Lists!

Unordered and Ordered Lists

Lists are super useful for organizing information:

  • Unordered Lists (<ul>): For items where order doesn't matter (e.g., shopping list). Each item is a <li>.
    • <ul><li>Item 1</li><li>Item 2</li></ul>
  • Ordered Lists (<ol>): For items where order matters (e.g., recipe steps). Each item is also a <li>.
    • <ol><li>Step 1</li><li>Step 2</li></ol>

Day 3: Semantic HTML

Giving Meaning to Your Code

Semantic HTML tags describe the meaning of content, not just how it looks. This helps browsers, search engines, and screen readers.

  • <header>: Introduction/navigation.
  • <nav>: Navigation links.
  • <main>: Dominant content.
  • <article>: Independent content (e.g., blog post).
  • <section>: Thematic grouping.
  • <aside>: Related but separate content (e.g., sidebar).
  • <footer>: Copyright, contact info.

Why? Improves accessibility and SEO!

Day 4: Connecting the Web with Links!

The Power of the Anchor Tag

Links are what make the "web" truly a web! We use the anchor tag: <a>

  • The href attribute specifies the destination URL.
  • The text between the tags is what the user clicks.
<a href="https://www.google.com">Go to Google</a>

Day 4: Types of Links

External, Internal, and Target Attributes

  • External Links: To another website (e.g., https://www.wikipedia.org).
  • Internal Links: To another page within your own website (e.g., about-me.html).
  • Target Attribute: Controls where the link opens:
    • _self (default): Same window/tab.
    • _blank: New window/tab (recommended for external links!).
  • Email Links: <a href="mailto:your.email@example.com">Email Me</a>

Day 5: Bringing Life to Your Page with Images!

The <img> Tag

The <img> tag embeds images. It's a self-closing tag.

  • src (source): Path to the image file.
  • alt (alternative text): Text description for accessibility and if the image fails to load.
<img src="my-picture.jpg" alt="A photo of me smiling">

Day 5: Image Paths and Sizing

Relative vs. Absolute Paths & Dimensions

  • Relative Paths: For images within your project (e.g., `images/my-pic.png`).
  • Absolute Paths (URLs): For images on other websites (e.g., `https://www.example.com/logo.png`).
  • Width and Height Attributes: Can set size in HTML (`width="300"`), but CSS is generally better for responsive design.

Day 6: Organizing Data with Tables!

Rows, Columns, and Cells

Tables display tabular data (rows and columns):

  • <table>: Container for the table.
  • <tr>: Table Row.
  • <th>: Table Header (bold, centered).
  • <td>: Table Data cell.

<table>
    <tr><th>Header</th><td>Data</td></tr>
</table>

Day 6: Table Enhancements

Captions, Headers, Bodies, and Footers

For better structure and accessibility:

  • <caption>: Table title.
  • <thead>, <tbody>, <tfoot>: Group table sections.
  • colspan: Cell spans multiple columns.
  • rowspan: Cell spans multiple rows.

<table>
    <caption>My Table</caption>
    <thead><tr><th>Col 1</th></tr></thead>
    <tbody><tr><td>Data</td></tr></tbody>
    <tfoot><tr><td colspan="2">Summary</td></tr></tfoot>
</table>

Day 7: Getting User Input with Forms! (Part 1)

The <form> and <input> Tags

Forms are essential for collecting information from users (e.g., login, contact forms, surveys).

  • <form>: The container for all form elements.
  • <label>: Provides a caption for an input element. Crucial for accessibility! Use the `for` attribute to link it to an input's `id`.
  • <input>: The most common form element. Its behavior changes based on the `type` attribute.

<form>
    <label for="username">Username:</label>
    <input type="text" id="username" name="username" placeholder="Enter your username">
    <button type="submit">Submit</button>
</form>

Day 7: Common Input Types

Text, Password, Email, Number

  • type="text": Single-line text input.
  • type="password": Text input where characters are masked (e.g., dots or asterisks).
  • type="email": For email addresses. Browsers can validate format.
  • type="number": For numerical input. Often shows up/down arrows.
  • placeholder attribute: Provides a hint to the user about what to enter.
  • name attribute: Important for identifying the data when the form is submitted.

<label for="user_email">Email:</label>
<input type="email" id="user_email" name="user_email" placeholder="you@example.com">

Day 8: More Form Elements! (Part 2)

Checkboxes, Radios, Textareas, and Dropdowns

  • type="checkbox": Allows users to select zero or more options.
  • type="radio": Allows users to select exactly one option from a group (use the same `name` attribute for a group).
  • <textarea>: For multi-line text input (e.g., comments). Has `rows` and `cols` attributes.
  • <select> and <option>: Creates a dropdown list.
  • required attribute: Makes an input field mandatory before form submission.

<input type="checkbox" id="html" name="skill" value="html">
<label for="html">HTML</label>

<input type="radio" id="male" name="gender" value="male">
<label for="male">Male</label>

<textarea id="comments" name="comments" rows="4" cols="50"></textarea>

<select id="country" name="country">
    <option value="usa">USA</option>
    <option value="can">Canada</option>
</select>

Day 8: Date, Color, and Range Inputs

Specialized Inputs for Specific Data

  • type="date": For selecting a date. Often provides a calendar picker.
  • type="color": For selecting a color. Provides a color picker.
  • type="range": For selecting a value from a range (e.g., a slider). Use `min`, `max`, `value`, and `step` attributes.

<label for="dob">Date of Birth:</label>
<input type="date" id="dob" name="dob">

<label for="favcolor">Favorite Color:</label>
<input type="color" id="favcolor" name="favcolor" value="#63b3ed">

<label for="volume">Volume:</label>
<input type="range" id="volume" name="volume" min="0" max="100" value="50">

Day 9: Bringing Media to Your Page!

Audio and Video Tags

HTML allows you to embed audio and video directly into your webpages!

  • <audio>: Embeds sound content.
  • <video>: Embeds video content.
  • Both tags use the `src` attribute for the media file path.
  • `controls` attribute: Adds default play/pause, volume, etc.
  • `autoplay`: Starts playing automatically (use with caution!).
  • `loop`: Repeats the media when it ends.

<audio src="mysong.mp3" controls loop>
    Your browser does not support the audio element.
</audio>

<video src="myvideo.mp4" controls width="400" height="225" poster="video-thumbnail.jpg">
    Your browser does not support the video tag.
</video>

The text inside the tags is fallback content for browsers that don't support the media.

Day 9: Multiple Media Sources & YouTube Embeds

Ensuring Compatibility & External Content

  • <source> tag: Inside `audio` or `video`, allows you to provide multiple file formats for broader browser compatibility. The browser will use the first one it supports.
    • 
<video controls>
    <source src="video.mp4" type="video/mp4">
    <source src="video.webm" type="video/webm">
    Your browser does not support the video tag.
</video>
  • Embedding YouTube/Vimeo: For popular video platforms, it's common to use their provided `<iframe>` embed code.
    • 
<iframe width="560" height="315" src="https://www.youtube.com/embed/dQw4w9WgXcQ" 
    frameborder="0" allowfullscreen></iframe>

Day 11: Making it Pretty with CSS! (Part 1)

Introduction to CSS: Styling Your Webpages

Welcome to CSS! If HTML is the structure, CSS (Cascading Style Sheets) is the designer of your webpage. It controls how your HTML elements look.

  • What colors should your text be?
  • How big should your headings be?
  • Where should elements be placed on the page?
  • What fonts should you use?

CSS answers all these questions and more!

Day 11: CSS Syntax

Selectors, Properties, and Values

A basic CSS rule has three main parts:


selector {
    property: value;
    property: value;
}
  • Selector: Points to the HTML element(s) you want to style (e.g., `p`, `h1`, `body`).
  • Declaration: The `property: value;` pair.
    • Property: The specific style attribute you want to change (e.g., `color`, `font-size`, `background-color`).
    • Value: The setting for that property (e.g., `blue`, `16px`, `#333`).

Example: `p { color: blue; font-size: 16px; }`

Day 11: Ways to Add CSS - Inline Styles

Directly on HTML Elements (Not Recommended for Most Cases!)

Inline styles are added directly to an HTML element using the `style` attribute.


<h1 style="color: blue; text-align: center;">Hello World!</h1>
<p style="font-size: 18px; background-color: lightgray;">This paragraph has inline styles.</p>

Pros: Quick for small, unique changes.
Cons: Hard to manage for large sites, mixes content and style, not reusable.

Day 11: Ways to Add CSS - Internal Styles

Inside the <head> of Your HTML Document

Internal styles are placed inside a <style> tag within the <head> section of your HTML document.


<head>
    <style>
        body {
            background-color: #1a202c;
            color: #e2e8f0;
        }
        h1 {
            color: #63b3ed;
            text-align: center;
        }
        p {
            font-family: sans-serif;
            font-size: 1.1rem;
        }
    </style>
</head>

Pros: Styles are in one place for that page.
Cons: Not reusable across multiple pages, can make HTML files very long.

Day 14: The CSS Box Model!

Content, Padding, Border, Margin

Every HTML element is treated as a rectangular box by the browser. Understanding the Box Model is crucial for layout and spacing.

  • Content: The actual content (text, image, video). Controlled by `width` and `height`.
  • Padding: Space between the content and the border. Pushes content away from the border.
  • Border: A line that goes around the padding and content.
  • Margin: Space outside the border. Pushes other elements away from this element.

div {
    width: 200px;
    height: 100px;
    padding: 20px; /* Space inside */
    border: 5px solid red; /* Line around */
    margin: 30px; /* Space outside */
}

Day 14: Padding, Border, Margin - Shorthands

Controlling Each Side & Collapsing Margins

  • Individual Sides: `padding-top`, `padding-right`, `padding-bottom`, `padding-left` (same for `margin`, `border`).
  • Shorthand (1 value): Applies to all 4 sides. 
    padding: 20px; /* 20px on all sides */
  • Shorthand (2 values): Top/Bottom, Left/Right. 
    padding: 10px 20px; /* 10px top/bottom, 20px left/right */
  • Shorthand (3 values): Top, Left/Right, Bottom. 
    padding: 10px 20px 30px; /* 10px top, 20px left/right, 30px bottom */
  • Shorthand (4 values): Top, Right, Bottom, Left (clockwise from top). 
    padding: 10px 20px 30px 40px;
  • Border Shorthand: `border: [width] [style] [color];` (e.g., `border: 2px solid #63b3ed;`)
  • Margin Collapsing: Vertical margins between block elements sometimes collapse into a single margin.

Day 15: Controlling Element Flow with `display`!

Block, Inline, and Inline-Block

The `display` property is fundamental to how elements behave on a page. Every HTML element has a default `display` value.

  • `display: block;`
    • Takes up the full width available.
    • Starts on a new line.
    • Can set `width`, `height`, `padding`, `margin`.
    • Examples: `<p>`, `<div>`, `<h1>`, `<li>`
  • `display: inline;`
    • Takes up only as much width as its content.
    • Does NOT start on a new line.
    • Cannot set `width` or `height`. `padding-top`/`bottom` and `margin-top`/`bottom` have no effect.
    • Examples: `<a>`, `<span>`, `<img>` (by default, but often styled as inline-block)
  • `display: inline-block;`
    • Behaves like an inline element (doesn't start new line).
    • But, you CAN set `width`, `height`, `padding`, and `margin`.
    • Useful for arranging elements side-by-side while giving them dimensions.

Day 16: Powerful Layouts with Flexbox! (Part 1)

Introduction to Flexbox: One-Dimensional Layout

Flexbox is a CSS layout module that makes it incredibly easy to arrange items in a single row or column. It's perfect for navigation bars, simple grids, and aligning content.

Flexbox works with two main concepts:

  • Flex Container: The parent element that holds the flex items. You apply `display: flex;` to this.
  • Flex Items: The direct children of the flex container.

<div class="container">
    <div class="item">Item 1</div>
    <div class="item">Item 2</div>
    <div class="item">Item 3</div>
</div>

/* CSS */
.container {
    display: flex; /* Makes this a flex container */
}

    • `column`: Items arranged vertically.
    • `row-reverse`, `column-reverse`: Reverses the order.
  • `justify-content`: Aligns items along themain axis.
    • `flex-start` (default), `flex-end`, `center`, `space-between`, `space-around`, `space-evenly`
  • `align-items`: Aligns items along the cross axis (perpendicular to main axis).
    • `stretch` (default), `flex-start`, `flex-end`, `center`, `baseline`

Day 17: Flexbox Item Properties! (Part 2)

Controlling Individual Flex Items

While container properties control the overall layout, flex item properties allow you to control individual children:

  • `flex-grow`: Defines the ability of a flex item to grow if necessary. Takes a unitless value (e.g., `1`, `2`). Items with higher `flex-grow` values take up more available space.
  • `flex-shrink`: Defines the ability of a flex item to shrink if necessary.
  • `flex-basis`: Defines the default size of an element before the remaining space is distributed. Can be a length (e.g., `100px`) or a percentage (`50%`).
  • `flex` shorthand: Combines `flex-grow`, `flex-shrink`, and `flex-basis` (e.g., `flex: 1 1 0;`).
  • `align-self`: Overrides the container's `align-items` for a single flex item.
  • `order`: Controls the order in which flex items appear in the flex container. Defaults to `0`.

.item-grow { flex-grow: 1; }
.item-specific-align { align-self: flex-end; }
.item-first { order: -1; } /* Appears first */

Day 17: Responsive Considerations with Flexbox

Wrapping and Responsiveness

  • `flex-wrap`: Controls whether flex items are forced onto one line or can wrap onto multiple lines.
    • `nowrap` (default): All items stay on one line, potentially overflowing.
    • `wrap`: Items wrap to the next line when they run out of space.
    • `wrap-reverse`: Items wrap in reverse order.
  • `flex-flow` shorthand: Combines `flex-direction` and `flex-wrap`.

Using `flex-wrap: wrap;` is crucial for creating responsive layouts where items stack or flow differently on smaller screens.


.responsive-container {
    display: flex;
    flex-wrap: wrap; /* Allows items to wrap to next line */
    justify-content: space-around;
}
.responsive-item {
    flex-basis: 300px; /* Default width */
    flex-grow: 1; /* Allows items to grow to fill space */
    margin: 10px;
}

Day 18: Mastering Element Positioning!

The `position` Property

The `position` property controls how an element is positioned on the page. Combined with `top`, `right`, `bottom`, and `left`, it offers precise control.

  • `position: static;` (Default): Elements are positioned according to the normal flow of the document. `top`, `right`, `bottom`, `left` have no effect.
  • `position: relative;`: Positions the element relative to its normal position. `top`, `right`, `bottom`, `left` values will shift it from where it *would* normally be. Other elements are not affected by this shift.
  • `position: absolute;`: Positions the element relative to its *nearest positioned ancestor* (an ancestor with `position` other than `static`). If no positioned ancestor, it's relative to the `<html>` element. It's removed from the normal document flow.
  • `position: fixed;`: Positions the element relative to the viewport. It stays in the same place even when the page is scrolled. Useful for sticky headers/footers.
  • `position: sticky;`: A mix of `relative` and `fixed`. Behaves relatively until a certain scroll point, then becomes fixed.

Day 18: Positioning Properties: `top`, `right`, `bottom`, `left`, `z-index`

Controlling Location and Stacking Order

  • `top`, `right`, `bottom`, `left`: These properties are used *with* `position` (except `static`) to specify the element's offset from its positioning context. 
    
.box {
    position: relative;
    top: 20px; /* Moves 20px down from its normal top */
    left: 10px; /* Moves 10px right from its normal left */
}
  • `z-index`: Controls the stacking order of positioned elements. Elements with a higher `z-index` value appear on top of elements with a lower value. Only works on positioned elements (`relative`, `absolute`, `fixed`, `sticky`). 
    
.element-on-top {
    position: absolute;
    z-index: 10; /* Will appear above elements with z-index less than 10 */
}

Day 25: Repeating Actions with Loops! (Part 1)

The `for` Loop

Loops are used to execute a block of code repeatedly as long as a certain condition is met. They are essential for tasks that involve iterating over collections of data or performing actions a fixed number of times.

  • **The `for` loop:** The most common type of loop, ideal when you know how many times you want to repeat an action. 
    
for (initialization; condition; afterthought) {
    // Code to be executed repeatedly
}

}

Iteration number: 1
Iteration number: 2
Iteration number: 3
Iteration number: 4
*/

Day 26: More Loops & Multi-Way Decisions!

`while` and `do...while` Loops, `switch` Statement

  • **`while` loop:** Repeats a block of code as long as a specified condition is `true`. The condition is checked *before* each iteration. 
    
let count = 0;
while (count < 5) {
    console.log("Count is: " + count);
    count++; // Important: increment to avoid infinite loop!
}
/* Output:
Count is: 0
Count is: 1
Count is: 2
Count is: 3
Count is: 4
*/
  • **`do...while` loop:** Similar to `while`, but the block of code is executed at least once, *then* the condition is checked. 
    
let i = 0;
do {
    console.log("Do-while count: " + i);
    i++;
} while (i < 3);
/* Output:
Do-while count: 0
Do-while count: 1
Do-while count: 2
*/
  • **`switch` statement:** A more efficient way to handle multiple `else if` conditions, especially when checking a single variable against many possible values. 
    
let day = "Tuesday";
switch (day) {
    case "Monday":
        console.log("Start of the week.");
        break; // Important: exits the switch block
    case "Tuesday":
    case "Wednesday": // Fall-through: if Tuesday or Wednesday, execute this
        console.log("Mid-week grind.");
        break;
    default: // If no case matches
        console.log("Weekend or unknown day.");
}

Day 26: When to Use Which Loop?

Choosing the Right Tool for the Job

  • **`for` loop:**
    • When you know the exact number of iterations.
    • When iterating over a sequence (e.g., numbers from 1 to 10).
  • **`while` loop:**
    • When the number of iterations is unknown, and the loop continues as long as a condition is true (e.g., reading data until the end of a file, or until a user enters valid input).
  • **`do...while` loop:**
    • When you need the loop body to execute at least once, regardless of the initial condition (e.g., prompting a user for input at least once).
  • **`switch` statement:**
    • When you have a single expression that you want to compare against multiple distinct values.
    • Often more readable than a long `if...else if...else` chain for specific value checks.

Functions are blocks of code designed to perform a particular task. They allow you to write reusable code, making your programs more organized, efficient, and easier to debug.

  • **Function Expression:** Defining a function as part of an expression, often assigned to a variable.

Day 30: Responding to User Actions (Events)!

`addEventListener` and Common Events

Websites become truly interactive when they respond to what users do (clicks, typing, mouse movements). These user actions are called **events**, and JavaScript allows you to "listen" for them and execute code when they occur.

  • **`element.addEventListener('eventName', functionName)`:** The most common and flexible way to handle events.
    • `eventName`: A string representing the type of event (e.g., `'click'`, `'mouseover'`, `'keydown'`).
    • `functionName` (or an anonymous function): The function that will be executed when the event occurs. This is called the **event handler** or **callback function**.
    
<button id="my-button">Click Me</button>
const myButton = document.getElementById('my-button');

myButton.addEventListener('click', function() {
    console.log("Button clicked!");
});

// Or with an arrow function:
myButton.addEventListener('mouseover', () => {
    console.log("Mouse is over the button!");
});
  • **The `event` Object:** The event handler function automatically receives an `event` object as its first argument. This object contains useful information about the event that occurred. 
    
myButton.addEventListener('click', function(event) {
    console.log("Event type:", event.type); // 'click'
    console.log("Target element:", event.target); // The button element
});

Day 30: Common Event Types

Interacting with Your Users

Here are some of the most frequently used event types:

  • **Mouse Events:**
    • `'click'`: When an element is clicked.
    • `'dblclick'`: When an element is double-clicked.
    • `'mouseover'`: When the mouse pointer enters an element.
    • `'mouseout'`: When the mouse pointer leaves an element.
    • `'mousedown'`, `'mouseup'`: When a mouse button is pressed/released over an element.
  • **Keyboard Events:**
    • `'keydown'`: When a key is pressed down.
    • `'keyup'`: When a key is released.
    • `'keypress'`: When a key is pressed and held down (less common now).
  • **Form Events:**
    • `'submit'`: When a form is submitted.
    • `'input'`: When the value of an `<input>` or `<textarea>` element changes.
    • `'change'`: When the value of an `<input>`, `<select>`, or `<textarea>` element changes and the element loses focus.
  • **Window/Document Events:**
    • `'load'`: When the entire page (including all resources like images) has loaded.
    • `'DOMContentLoaded'`: When the HTML document has been completely loaded and parsed, without waiting for stylesheets, images, and subframes to finish loading. (Often preferred over `'load'`).
    • `'resize'`: When the browser window is resized.

Day 38: Understanding Asynchronous JavaScript!

Non-Blocking Code & Callbacks

So far, your JavaScript has been **synchronous**: code executes line by line, and each line must finish before the next one starts. But what if you need to fetch data from the internet, which takes time?

**Asynchronous JavaScript** allows certain operations to run in the background without blocking the main execution thread. This means your website remains responsive while waiting for long-running tasks to complete.

  • Blocking vs. Non-Blocking:
    • **Blocking (Synchronous):** Code runs sequentially. If one task takes a long time, the entire program "freezes" until it's done.
    • **Non-Blocking (Asynchronous):** Long-running tasks are initiated, and then the program continues executing other code. When the long task finishes, it signals that it's done and its result is handled.
  • Callbacks: A function that is passed as an argument to another function and is executed later, often after an asynchronous operation completes. `setTimeout` is a great example!

console.log("Start");

setTimeout(() => {
    console.log("This runs after 1 second (asynchronously)");
}, 1000);

console.log("End (This runs immediately after Start)");

/* Output:
Start
End (This runs immediately after Start)
This runs after 1 second (asynchronously)
*/

Day 38: Why Asynchronous JS is Important

Keeping Your User Interface Responsive

Imagine if your website froze every time it needed to load an image or fetch data from a server! Asynchronous JavaScript prevents this, ensuring a smooth user experience.

  • **Network Requests:** Fetching data from APIs (Application Programming Interfaces) is a prime example of an asynchronous operation.
  • **File I/O:** Reading or writing files (in Node.js environments).
  • **Timers:** `setTimeout` and `setInterval` are asynchronous.
  • **User Interactions:** Event listeners are asynchronous; the code in the event handler runs only when the event occurs.

Understanding asynchronous patterns is critical for building modern web applications. We'll start with callbacks today and move to more modern approaches (`Promises`, `async/await`) later in the camp.

Day 41: Advanced Array Methods - Filtering & Finding!

`filter()` and `find()`

JavaScript arrays come with many powerful built-in methods that make working with data much easier and more efficient than traditional loops.

  • `array.filter(callbackFunction)`: Creates a **new array** containing all elements that pass a test (i.e., for which the `callbackFunction` returns `true`). It does not modify the original array. 
    
const numbers = [1, 2, 3, 4, 5, 6];
const evenNumbers = numbers.filter(num => num % 2 === 0);
console.log(evenNumbers); // Output: [2, 4, 6]
  • `array.find(callbackFunction)`: Returns the **first element** in the array that satisfies the provided testing function. If no elements satisfy the testing function, `undefined` is returned. 
    
const students = [
    { name: "Alice", grade: 85 },
    { name: "Bob", grade: 92 },
    { name: "Charlie", grade: 88 }
];
const topStudent = students.find(student => student.grade > 90);
console.log(topStudent); // Output: { name: "Bob", grade: 92 }

Day 41: Why Use These Methods?

Cleaner, More Readable Code

These methods are part of "functional programming" paradigms in JavaScript. They offer several advantages:

  • **Readability:** The code often expresses its intent more clearly than a traditional `for` loop.
  • **Conciseness:** You can achieve complex operations with fewer lines of code.
  • **Immutability:** `filter()` creates a new array, which is a good practice as it avoids modifying the original data structure directly.

The callback function for `filter()` and `find()` receives three arguments: `(element, index, array)`. You usually only need the `element`.

Day 44: Diving Deeper into Promises!

States, Chaining, and `Promise.all()`

We briefly touched on Promises with the Fetch API. Today, we'll understand them more thoroughly. A **Promise** is an object representing the eventual completion or failure of an asynchronous operation.

  • Promise States:
    • `pending`: Initial state, neither fulfilled nor rejected.
    • `fulfilled` (or `resolved`): The operation completed successfully.
    • `rejected`: The operation failed.
  • Creating a Promise: 
    
const myPromise = new Promise((resolve, reject) => {
    // Do an asynchronous operation here (e.g., network request, timer)
    const success = true; // Simulate success/failure
    if (success) {
        resolve("Operation successful!"); // Call resolve on success
    } else {
        reject("Operation failed!"); // Call reject on failure
    }
});

myPromise
    .then(message => console.log(message)) // Handles success
    .catch(error => console.error(error)); // Handles failure
  • Promise Chaining: When one asynchronous operation depends on the result of another, you can chain `.then()` calls. 
    
fetch('url1')
    .then(response => response.json())
    .then(data1 => {
        console.log(data1);
        return fetch('url2?id=' + data1.id); // Return another fetch promise
    })
    .then(response => response.json())
    .then(data2 => console.log(data2))
    .catch(error => console.error(error));

Day 44: `Promise.all()` and `Promise.race()`

Handling Multiple Promises

  • `Promise.all(iterable)`: Takes an iterable of Promises as input and returns a single Promise that resolves when all of the input Promises have resolved, or rejects when any of the input Promises rejects. 
    
const promise1 = Promise.resolve(3);
const promise2 = 42; // Non-promise values are treated as resolved promises
const promise3 = new Promise((resolve, reject) => {
    setTimeout(resolve, 100, 'foo');
});

Promise.all([promise1, promise2, promise3])
    .then(values => {
        console.log(values); // Output: [3, 42, "foo"]
    })
    .catch(error => console.error(error));
  • `Promise.race(iterable)`: Returns a Promise that resolves or rejects as soon as one of the Promises in the iterable resolves or rejects, with the value or reason from that Promise. 
    
const p1 = new Promise(resolve => setTimeout(resolve, 500, 'one'));
const p2 = new Promise(resolve => setTimeout(resolve, 100, 'two'));

Promise.race([p1, p2])
    .then(value => {
        console.log(value); // Output: "two" (p2 resolves faster)
    });

Day 45: Simplifying Async Code with Async/Await!

Cleaner Asynchronous Syntax

**`async/await`** is a modern JavaScript feature built on top of Promises that makes asynchronous code look and behave more like synchronous code, making it much easier to read and write.

  • `async` keyword: You must put `async` before a function declaration to indicate that the function will perform asynchronous operations and will implicitly return a Promise. 
    
async function fetchData() {
    // ... asynchronous code ...
}
  • `await` keyword: Can only be used inside an `async` function. It pauses the execution of the `async` function until the Promise it's waiting for settles (resolves or rejects). 
    
async function getQuote() {
    try {
        const response = await fetch('https://api.quotable.io/random');
        const data = await response.json();
        console.log(data.content);
    } catch (error) {
        console.error("Failed to fetch quote:", error);
    }
}
getQuote();

Notice how `try...catch` is used with `async/await` for error handling, similar to synchronous code.

Day 45: Benefits of `async/await`

Readability and Error Handling

While `async/await` is syntactic sugar over Promises (meaning it's just a different way to write the same underlying logic), it offers significant advantages:

  • **Improved Readability:** Code flows from top to bottom, resembling synchronous code, making it easier to follow the logic.
  • **Simpler Error Handling:** You can use standard `try...catch` blocks to handle errors from asynchronous operations, just like synchronous errors. This avoids multiple `.catch()` blocks in long Promise chains.
  • **Debugging:** Debugging `async/await` code can sometimes be more straightforward than debugging complex Promise chains.

`async/await` is the preferred way to write asynchronous JavaScript in modern applications.

Parallel Async Operations & Robust Error Handling

When dealing with multiple asynchronous operations, especially those that don't depend on each other, you can run them in parallel to improve performance.

async function fetchAllData() {
    try {
        const [usersResponse, postsResponse] = await Promise.all([
            fetch('https://jsonplaceholder.typicode.com/users'),
            fetch('https://jsonplaceholder.typicode.com/posts')

        const posts = await postsResponse.json();

        console.log("Posts:", posts.length);

        console.error("Error fetching multiple data:", error);

}
fetchAllData();
  • Error Handling in Parallel: If any Promise in `Promise.all()` rejects, the entire `Promise.all()` will reject, and the `catch` block will be executed.

Day 47: Making Your Website Responsive! (Part 1)

Media Queries: Adapting to Screen Sizes

**Responsive Web Design** is about making your website look good and function well on all devices, from small phones to large desktop monitors. **Media Queries** are a core CSS feature for achieving this.

  • Media queries allow you to apply CSS styles only when certain conditions are met, such as screen width, height, or orientation.
  • They use the `@media` rule.

/* Styles for screens smaller than 600px wide */
@media (max-width: 599px) {
    body {
        font-size: 16px;
    }
    .container {
        flex-direction: column; /* Stack items vertically on small screens */
    }
}

/* Styles for screens 600px wide and larger */
@media (min-width: 600px) {
    body {
        font-size: 18px;
    }
    .container {
        flex-direction: row; /* Keep items horizontal on larger screens */
    }
}

Common breakpoints are often around 640px (small phones/tablets), 768px (tablets), 1024px (laptops), 1280px (desktops).

Day 47: Viewport Meta Tag & Mobile-First Approach

Essential for Mobile Responsiveness

  • Viewport Meta Tag: This HTML meta tag is absolutely crucial for responsive design. It tells the browser to set the width of the viewport to the device's width and set the initial zoom level. Without it, mobile browsers might render your page at a desktop width and then zoom out. 
    <meta name="viewport" content="width=device-width, initial-scale=1.0">

    Always include this in your `<head>`!

  • Mobile-First Approach:
    • Design and style your website for small screens (mobile) first.
    • Then, use media queries with `min-width` to add styles for larger screens.
    • This ensures a solid base for mobile users and progressively enhances for larger devices. It's often easier than trying to "undo" desktop styles for mobile.
    
/* Mobile styles (default) */
.my-element {
    width: 100%;
    padding: 10px;
}

/* Tablet styles */
@media (min-width: 768px) {
    .my-element {
        width: 50%; /* Takes half width on tablets */
    }
}

/* Desktop styles */
@media (min-width: 1024px) {
    .my-element {
        width: 33%; /* Takes one-third width on desktops */
    }
}

Day 49: Powerful 2D Layouts with CSS Grid! (Introduction)

Rows, Columns, and Grid Areas

**CSS Grid Layout** is a two-dimensional layout system for the web. It's designed for laying out major page regions or small user interface elements in rows and columns, making complex layouts much simpler than with Flexbox (which is one-dimensional).

  • Grid Container: The parent element to which you apply `display: grid;`.
  • Grid Items: The direct children of the grid container.
  • Grid Lines: The horizontal and vertical lines that divide the grid.
  • Grid Tracks: The space between two grid lines (rows or columns).
  • Grid Cells: The intersection of a row and a column.
  • Grid Areas: Named areas spanning multiple cells.

/* HTML */
<div class="grid-container">
    <div class="header">Header</div>
    <div class="sidebar">Sidebar</div>
    <div class="main">Main Content</div>
    <div class="footer">Footer</div>
</div>

/* CSS */
.grid-container {
    display: grid;
    grid-template-columns: 1fr 3fr; /* Two columns: 1 fractional unit, 3 fractional units */
    grid-template-rows: auto 1fr auto; /* Header auto height, main 1 fractional unit, footer auto */
    gap: 20px; /* Space between grid items */
}

Day 49: Defining Grid Structure & Item Placement

`grid-template-columns`, `grid-template-rows`, `grid-area`

  • `grid-template-columns` / `grid-template-rows`: Defines the number and size of columns/rows.
    • `px`, `%`, `em`, `rem`: Fixed sizes.
    • `fr` (fractional unit): Takes up a fraction of the available space.

Day 50: Storing Data in the Browser!

`localStorage` and `sessionStorage`

**Web Storage** allows web applications to store data locally within the user's browser. This is useful for saving user preferences, shopping cart items, or temporary data without needing a server.

  • `localStorage`: Stores data with no expiration date. The data persists even when the browser window is closed and reopened.
    • `localStorage.setItem('key', 'value')`: Stores a key-value pair. Values are always stored as strings.
    • `localStorage.getItem('key')`: Retrieves the value for a given key.
    • `localStorage.removeItem('key')`: Removes a key-value pair.
    • `localStorage.clear()`: Clears all items from local storage for the current domain.
    
localStorage.setItem('username', 'coder_user');
const savedUser = localStorage.getItem('username');
console.log(savedUser); // Output: "coder_user"
  • `sessionStorage`: Stores data for the duration of a single browser session. Data is cleared when the browser tab/window is closed. Syntax is identical to `localStorage`. 
    
sessionStorage.setItem('temp_data', 'This is temporary');

Converting to and from Strings

Since Web Storage only stores strings, you need to convert JavaScript objects and arrays to JSON strings before storing them, and then parse them back into JavaScript objects/arrays when retrieving.

const userSettings = { theme: "dark", notifications: true };

const retrievedSettingsString = localStorage.getItem('settings');

console.log(retrievedSettings.theme); // Output: "dark"

Day 51: Sending Data to the Server with Fetch API!

`POST` Requests and Request Headers

So far, we've used `fetch` to `GET` data. Now, let's learn how to `POST` (send) data to a server, which is essential for forms, creating new resources, etc.

  • `fetch(url, options)`: The second argument, `options`, is an object where you define the request method, headers, and body.
  • `method`: Specifies the HTTP method (e.g., `'POST'`, `'PUT'`, `'DELETE'`). Default is `'GET'`.
  • `headers`: An object containing HTTP headers. Crucial for telling the server what type of content you're sending (e.g., `Content-Type: application/json`).
  • `body`: The data you want to send. For JSON data, you must `JSON.stringify()` your JavaScript object.

async function postData() {
    const dataToSend = { title: 'My New Post', body: 'This is the content.', userId: 1 };

    try {
        const response = await fetch('https://jsonplaceholder.typicode.com/posts', {
            method: 'POST', // Specify POST method
            headers: {
                'Content-Type': 'application/json', // Tell server we're sending JSON
            },
            body: JSON.stringify(dataToSend), // Convert JS object to JSON string
        });

        if (!response.ok) {
            throw new Error(`HTTP error! status: ${response.status}`);
        }

        const result = await response.json();
        console.log('Success:', result); // Server often returns the created resource
    } catch (error) {
        console.error('Error:', error);
    }
}
postData();

Day 51: Common HTTP Status Codes

Understanding Server Responses

When you make an HTTP request, the server responds with a status code indicating the outcome. Understanding these helps with debugging and error handling.

  • **Informational responses (1xx):** Request received, continuing process.
  • **Successful responses (2xx):**
    • `200 OK`: Standard success for `GET`, `PUT`, `POST`.
    • `201 Created`: Resource successfully created (common for `POST`).
    • `204 No Content`: Request successful, but no content to return (common for `DELETE`).
  • **Redirection messages (3xx):** Further action needs to be taken.
  • **Client error responses (4xx):**
    • `400 Bad Request`: Server cannot process request due to client error (e.g., malformed syntax).
    • `401 Unauthorized`: Authentication required.
    • `403 Forbidden`: Client does not have access rights.
    • `404 Not Found`: Server cannot find the requested resource.
  • **Server error responses (5xx):**
    • `500 Internal Server Error`: Generic server error.
    • `503 Service Unavailable`: Server is not ready to handle the request.

Day 53: Modern JavaScript Features (ES6+)!

Modules (`import`/`export`) & Advanced Spread/Rest

JavaScript is constantly evolving. ES6 (ECMAScript 2015) introduced many features that are now standard. We've already used some (like `let`, `const`, arrow functions, template literals). Let's explore more!

  • JavaScript Modules (`import`/`export`): Allows you to break your JavaScript code into smaller, reusable files. This makes your code more organized, maintainable, and prevents naming conflicts.
    • `export`: Used to export functions, variables, or classes from a module. 
      
// utils.js
export const PI = 3.14;
export function add(a, b) { return a + b; }
export default function subtract(a, b) { return a - b; } // Only one default export per module
    • `import`: Used to import exported members into another module. 
      
// main.js
import { PI, add } from './utils.js'; // Named imports
import subtract from './utils.js'; // Default import
console.log(add(5, 2)); // Output: 7
console.log(subtract(10, 3)); // Output: 7
    • To use modules in HTML, add `type="module"` to your script tag: 
      <script type="module" src="main.js"></script>
  • Rest Parameter (`...` in function arguments): Collects an indefinite number of arguments into an array. 
    
function sumAll(...numbers) {
    return numbers.reduce((acc, num) => acc + num, 0);
}
console.log(sumAll(1, 2, 3, 4)); // Output: 10

Day 53: Destructuring in Function Parameters & Optional Chaining

Cleaner Function Signatures & Safer Property Access

  • Destructuring in Function Parameters: Extract properties from objects (or values from arrays) directly in the function's parameter list. 
    
function displayUser({ name, email }) { // Destructures 'name' and 'email' from the passed object

}
const user = { name: "Jane Doe", age: 25, email: "jane@example.com" };
displayUser(user);
  • Optional Chaining (`?.`): A safe way to access nested object properties without causing an error if an intermediate property is `null` or `undefined`. 
    
const user = {
    name: "Alice",
    address: {
        street: "123 Main St"

};
console.log(user.address?.city);     // Output: undefined (no error)

Day 56: Becoming a Debugging Detective!

Effective Debugging Strategies & Browser DevTools

Writing code is only half the battle; the other half is fixing when things go wrong! **Debugging** is the process of finding and fixing errors (bugs) in your code. Mastering debugging saves you a lot of time and frustration.

  • The Browser Developer Tools: Your indispensable tool for debugging HTML, CSS, and JavaScript.
    • **Elements Tab:** Inspect and modify HTML structure and applied CSS styles in real-time.
    • **Console Tab:** View `console.log()` messages, error messages, and run JavaScript commands directly.
    • **Sources Tab:** Set breakpoints, step through your JavaScript code line by line, inspect variable values.
    • **Network Tab:** Monitor network requests (Fetch API calls), see request/response details, status codes.
  • Common Debugging Strategies:
    • **`console.log()`:** The simplest way to check variable values at different points in your code.
    • **Breakpoints:** Pause code execution at a specific line in the Sources tab. Then, you can:
      • **Step Over:** Execute the current line and move to the next.
      • **Step Into:** If the current line calls a function, step into that function's code.
      • **Step Out:** If inside a function, execute the rest of the function and return to the calling code.
      • **Resume Script Execution:** Continue running the code until the next breakpoint or end of script.
    • **Inspect Elements:** Use the Elements tab to see if your HTML structure is as expected and if CSS rules are being applied correctly (and why they might not be).
    • **Isolate the Problem:** Comment out sections of code or remove elements to narrow down where the bug is.
    • **Read Error Messages:** The console provides valuable clues about the type of error and where it occurred (file and line number).

Day 57: Writing Clean, Maintainable Code!

Code Organization & Best Practices

Writing code that works is great, but writing code that is easy to read, understand, and maintain (by yourself or others!) is even better. This is where **code organization** and **best practices** come in.

  • Naming Conventions:
    • **Variables/Functions:** `camelCase` (e.g., `myVariable`, `calculateSum`).
    • **Constants:** `UPPER_SNAKE_CASE` (e.g., `MAX_VALUE`, `API_KEY`).
    • **Classes (JS):** `PascalCase` (e.g., `MyComponent`, `ProjectCard`).
    • **CSS Classes:** `kebab-case` (e.g., `main-header`, `project-card`).
    • Be descriptive! `const userAge` is better than `const ua`.
  • Comments: Explain *why* you're doing something, not just *what* you're doing. Keep them concise and up-to-date. 
    
// Calculate total price including tax
// This function assumes a 10% tax rate
function calculateTotalPrice(price) {
    return price * 1.10;
}
  • Indentation & Formatting: Consistent indentation (2 or 4 spaces) makes code readable. Use linters (like ESLint - conceptual) to enforce this automatically.
  • Modularity: Break code into smaller, focused functions and modules (`.js` files). Each function/module should ideally do one thing well.
  • DRY Principle (Don't Repeat Yourself): If you find yourself writing the same code multiple times, consider putting it into a reusable function.
  • Avoid Global Variables: Limit the use of global variables as they can lead to naming conflicts and make code harder to reason about. Use `const` and `let` within functions or modules.
  • Error Handling: Implement `try...catch` for robust code, especially for asynchronous operations or user input.

Day 57: Refactoring Code

Improving Existing Code Without Changing Functionality

**Refactoring** is the process of restructuring existing computer code without changing its external behavior. Its goal is to improve nonfunctional attributes of the software, such as readability, maintainability, and complexity.

  • Why Refactor?
    • Make code easier to understand.
    • Make code easier to modify and extend.
    • Reduce the likelihood of introducing new bugs.
    • Improve performance (sometimes, but not the primary goal).
  • Common Refactoring Techniques:
    • **Extract Function:** Turn a block of code into a separate function.
    • **Rename Variable/Function:** Use more descriptive names.
    • **Replace Magic Number with Named Constant:** Use `const PI = 3.14;` instead of `3.14` directly.
    • **Consolidate Conditional Expression:** Combine multiple `if` statements if possible.
    • **Remove Dead Code:** Delete unused variables, functions, or code blocks.

Day 58: Accessibility Considerations (Basic)

Making Your Website Usable for Everyone

**Web accessibility** means designing and developing websites so that people with disabilities can use them. Even basic considerations make a big difference.

  • **Semantic HTML:** Using tags like `<header>`, `<nav>`, `<main>`, `<footer>`, `<button>`, `<label>` (with `for` attribute) helps screen readers understand your page structure.
  • **`alt` attributes for Images:** Always provide descriptive `alt` text for all images. This is crucial for visually impaired users.
  • **Color Contrast:** Ensure sufficient contrast between text and background colors. Tools are available online to check this.
  • **Keyboard Navigation:** Make sure all interactive elements (links, buttons, form fields) can be accessed and operated using only the keyboard (use the `Tab` key to navigate).
  • **Clear Form Labels:** Use `<label>` tags correctly linked to their inputs.

Building accessible websites is a sign of a thoughtful and professional developer!

Day 59: Sharing Your Website: Deployment!

What is Deployment? Static Hosting Concepts

**Deployment** is the process of making your website accessible to others on the internet. Up until now, your website has only been visible on your local computer. To share it, you need to "host" it on a web server.

  • **Web Server:** A computer program that stores your website files (HTML, CSS, JS, images) and delivers them to users' browsers when they request them.
  • **Hosting:** The service that provides the web server and keeps your website online.
  • **Domain Name:** The human-readable address of your website (e.g., `google.com`, `yourname.com`).
  • **Static Hosting:** The simplest type of hosting, perfect for websites built only with HTML, CSS, and JavaScript (like your portfolio). The server just sends your files as-is.
  • **Popular Static Hosting Services:**
    • **GitHub Pages:** Free, integrates directly with GitHub repositories. Great for portfolios.
    • **Netlify:** Offers free tiers, very easy to use for static sites, integrates with Git.
    • **Vercel:** Similar to Netlify, also very popular for static sites and frontend projects.

Day 59: Preparing for Deployment (Conceptual)

File Structure & GitHub Basics

While we won't go through the full deployment process step-by-step in this camp (it involves setting up accounts and potentially Git), understanding the steps is key.

  • **Organize Your Files:** Ensure all your HTML, CSS, JavaScript, and image files are in a single, well-structured folder (your `my-portfolio` folder).
  • **`index.html` is Key:** Your main home page file *must* be named `index.html` (lowercase). This is the file web servers look for by default.
  • **Relative Paths:** Double-check that all your links (`<a href="...">`), image sources (`<img src="...">`), and CSS/JS links (`<link href="...">`, `<script src="...">`) use **relative paths** (e.g., `./style.css`, `./images/my-pic.jpg`, `./about.html`). Absolute paths (starting with `http://` or `/`) can cause issues if not handled carefully.
  • **Git & GitHub (Conceptual):**
    • **Git:** A version control system that tracks changes to your code.
    • **GitHub:** A platform for hosting Git repositories. Many hosting services (like GitHub Pages, Netlify) integrate directly with GitHub.
    • You would typically upload your entire `my-portfolio` folder to a GitHub repository.
